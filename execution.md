## Recommended First Module & Detailed Plan

### Recommendation
Build the **Users / Auth** module first (Firebase-backed user sync and basic profile routes). Rationale:
- Every protected route depends on user identity and role — building auth unlocks all other modules.
- It's low-risk, quick to iterate, and enables end-to-end verification with the DB and Prisma client.
- Once auth is stable, you can safely add quota, RBAC, and AI endpoints that rely on `req.user`.

### High-level objectives
- Initialize `firebase-admin` server SDK and load credentials from environment.
- Implement a robust `auth.middleware.ts` that verifies Firebase ID tokens and attaches `uid` and `email` to `req.user`.
- Implement `POST /api/auth/sync` to upsert a user record in Postgres using Prisma (id = Firebase UID).
- Implement `GET /api/users/me`, `PATCH /api/users/me`, and `DELETE /api/users/me` with correct FK cascading behavior.

### File-by-file implementation plan (ordered)
1. `src/config/firebase.ts`
  - Initialize `firebase-admin` from `FIREBASE_PRIVATE_KEY`, `FIREBASE_CLIENT_EMAIL`, and `FIREBASE_PROJECT_ID`.
  - Export a `verifyIdToken(token: string)` helper that calls `admin.auth().verifyIdToken(token)`.

2. `src/middleware/auth.middleware.ts`
  - Extract `Authorization: Bearer <token>` header.
  - Call `verifyIdToken` and attach `{ uid, email }` to `req.user`.
  - Return `401` on missing/invalid token.

3. `src/modules/users/users.service.ts`
  - `syncUserFromFirebase(payload)` → `prisma.users.upsert({ where: { id: uid }, create: {...}, update: {...} })`.
  - `getUserById(uid)`, `updateUser(uid, data)`, `deleteUser(uid)` (delete should cascade: delete CV files first then DB records or use `ON DELETE CASCADE`).

4. `src/modules/users/users.controller.ts`
  - `POST /api/auth/sync` — accepts no body; uses `req.user` after `auth.middleware` to upsert and return user.
  - `GET /api/users/me` — returns current user profile.
  - `PATCH /api/users/me` — accepts `name`, `target_role`, `experience_level` (use Zod schema), returns updated profile.
  - `DELETE /api/users/me` — deletes user data (warn: destructive).

5. `src/modules/users/users.routes.ts`
  - Wire controller methods to routes and export router. Protect routes with `auth.middleware`.

6. `src/app.ts` changes
  - Uncomment and mount `authRoutes` / `users` routes at `/api/auth` and `/api/users`.

7. Tests & verification
  - Unit test `users.service.syncUserFromFirebase` with a mock Prisma client.
  - Integration test: `POST /api/auth/sync` using a real `.env` DB (or test DB). Use a mocked valid Firebase token (or bypass verification in test by swapping middleware).
  - Verify `prisma` upsert creates user with `id = Firebase UID` and `email` persisted.

### Minimal request/response examples
- POST /api/auth/sync (Authorization header required)
  - Response: `{ success: true, user: { id, email, name, role } }`
- GET /api/users/me
  - Response: `{ success: true, user: { id, email, name, target_role, experience_level } }`

- Implement basic happy-path first (no deletions). Ship `auth.sync`, `GET /me`, and `PATCH /me`.
- Add `DELETE /me` once CV file deletion strategy is implemented or when storage is ready.

### Risks & mitigations
- Risk: malformed `FIREBASE_PRIVATE_KEY` in env (newlines). Mitigation: document required quoting in `.env.example`.
- Risk: DB constraint errors during delete. Mitigation: implement cascade deletes or explicit order.
